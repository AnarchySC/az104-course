Lab 06 — Implement Network Traffic Management

Estimated time: 50 minutes

1

Deploy Infrastructure via Template

▼

Deploy the lab infrastructure using an ARM template that creates one virtual network with three subnets, each containing a virtual machine.

  1. Navigate to Deploy a custom template in the Azure Portal.
  2. Select Build your own template in the editor.
  3. Load the template file az104-06-vms-template.json and click Save.
  4. Click Edit parameters and load the corresponding parameters file, then click Save.
  5. Configure deployment settings:
    SettingValue
    Resource Groupaz104-rg6 (create new)
    RegionEast US
  6. Click Review + create, then Create.
  7. Wait for deployment to complete. Verify the resource group contains:
    • 1 VNet with 3 subnets
    • 3 virtual machines (one per subnet)
    • Associated NICs, disks, and public IPs
Note The template deploys VMs with IIS installed via a custom script extension. This allows you to test load balancing by browsing to the backend VMs and seeing their hostnames in the "Hello World" response.
2

Configure Azure Load Balancer

▼

Create a public Azure Load Balancer to distribute Layer 4 traffic across two backend VMs.

  1. Navigate to Load balancers and click + Create.
  2. Configure the load balancer:
    SettingValue
    Resource Groupaz104-rg6
    Nameaz104-lb
    SKUStandard
    TypePublic
    TierRegional
  3. Configure the Frontend IP:
    SettingValue
    Nameaz104-fe
    Public IPaz104-lbpip (create new, static)
  4. Configure the Backend pool:
    SettingValue
    Nameaz104-be
    Virtual network(select lab VNet)
    Membersvm0, vm1
  5. Configure the Load balancing rule:
    SettingValue
    Nameaz104-lbrule
    Frontend IPaz104-fe
    Backend poolaz104-be
    Port80
    Backend port80
    Health probeaz104-hp (create new)
  6. Configure the Health probe:
    SettingValue
    Nameaz104-hp
    ProtocolTCP
    Port80
    Interval5 seconds
  7. Click Review + create, then Create.
  8. After deployment, navigate to the load balancer and note the Frontend public IP address.
  9. Open the frontend IP in a browser. You should see a "Hello World" page displaying the VM hostname.
  10. Refresh several times — observe the hostname alternating between vm0 and vm1, confirming load balancing.
Tip If you always see the same VM, try using an incognito/private window or clearing your browser cache. Session persistence settings can affect which backend serves each request.
3

Configure Azure Application Gateway

▼

Create an Application Gateway with path-based routing to direct HTTP traffic to different backend pools based on URL paths.

  1. First, create a new subnet for the Application Gateway:
    SettingValue
    Subnet namesubnet-appgw
    Address range10.60.3.224/27
  2. Navigate to Application Gateways and click + Create.
  3. Configure the basics:
    SettingValue
    Nameaz104-appgw
    Resource Groupaz104-rg6
    TierStandard V2
    AutoscalingNo
    Instance count2
    Subnetsubnet-appgw
  4. Configure the Frontend:
    SettingValue
    Frontend IP typePublic
    Public IPaz104-gwpip (create new)
  5. Add Backend pools:
    Pool NameMembers
    az104-appgwbevm1, vm2
    az104-imagebevm1
    az104-videobevm2
  6. Configure a Routing rule with path-based routing:
    SettingValue
    Rule nameaz104-routing
    ListenerHTTP on port 80
    Default backend poolaz104-appgwbe
    Path: /image/*az104-imagebe
    Path: /video/*az104-videobe
  7. Click Review + create, then Create. Deployment may take 15–20 minutes.
  8. After deployment, note the Application Gateway's Frontend public IP.
  9. Test the path-based routing in your browser:
    URL http://<gateway-ip>/image/ http://<gateway-ip>/video/ http://<gateway-ip>/
  10. Verify that each path routes to the expected backend pool.
Warning Application Gateway requires a dedicated subnet with no other resources. The subnet must be large enough for the instance count. A /27 (32 addresses) is the minimum recommended size.

Key Takeaways

  • Azure Load Balancer distributes traffic at Layer 4 (TCP/UDP) using hash-based distribution across backend pool members.
  • Azure Application Gateway manages traffic at Layer 7 (HTTP/HTTPS) with URL path-based routing and content-aware features.
  • Public load balancers distribute internet traffic; internal load balancers distribute traffic within a VNet.
  • Path-based routing enables directing requests to content-specific backend pools based on URL patterns.
  • The WAF tier on Application Gateway adds protection against common web vulnerabilities and malicious traffic filtering.

Load Balancer Creation and Configuration

▼

  1. Use the "Help me choose" wizard to understand which load balancing solution fits your scenario.
  2. Explain the SKU options: Basic (free, limited features) vs Standard (SLA-backed, zone-redundant).
  3. Explain the Type options: Public (internet-facing) vs Internal (private VNet traffic).

  5. Configure the Frontend IP — show the difference between static and dynamic allocation.
  6. Add VMs to a Backend pool and explain NIC-based vs IP-based configuration.
  7. Create Inbound rules: load balancing rules, inbound NAT rules, and outbound rules.
  8. Configure a Health probe and explain how unhealthy VMs are automatically removed from rotation.
Tip Standard SKU load balancers are secure by default — backend pool members must have an NSG allowing traffic, unlike Basic SKU which allows all traffic by default.

Application Gateway Configuration

▼

Demonstrate the Application Gateway setup and its Layer 7 capabilities:

  1. Explain the Tier options: Standard V2 (routing only) vs WAF V2 (routing + web application firewall).
  2. Show Frontend configuration: public IP, private IP, or both.
  3. Configure Backend pools with different target types: VMs, VMSS, App Services, IP addresses, or FQDN.
  4. Create HTTP Settings: port, protocol, cookie-based affinity, connection draining, request timeout.
  5. Configure Listeners: basic (single site) vs multi-site listeners.
  6. Set up Routing rules: basic routing vs path-based routing.
  7. Demonstrate path-based routing with multiple backend pools for different URL paths.
Note Application Gateway V2 supports autoscaling, zone redundancy, and static VIP — all improvements over V1. New deployments should always use V2.

Exercises

Easy

Exercise 1: Load Balancer Health Check

  1. Navigate to an existing Load Balancer in the Azure Portal.
  2. Check the Backend pool health status.
  3. Review the health probe configuration: protocol, port, and interval.
  4. Identify: What happens to traffic if one backend VM goes down?
Show Hint
The health probe continuously checks backend health at the configured interval. When a VM fails the probe threshold, it is automatically removed from rotation. Traffic is distributed only to healthy VMs until the unhealthy VM recovers and passes probes again.
Medium

Exercise 2: Load Balancing Comparison

  1. Review Azure's "Load balancing — help me choose" tool in the portal.
  2. Compare: When would you use Load Balancer vs Application Gateway vs Traffic Manager vs Front Door?
  3. Create a table documenting: OSI layer, protocol support, global vs regional, and key features for each service.
  4. For your current lab setup, justify which load balancing solution fits best.
Show Hint
Load Balancer = Layer 4 regional (TCP/UDP hash-based distribution). Application Gateway = Layer 7 regional (HTTP path-based routing, SSL termination, WAF). Traffic Manager = DNS-based global (geographic/performance/priority routing). Front Door = Layer 7 global (CDN + WAF + global load balancing + SSL offload).
Hard

Exercise 3: Path-Based Routing Design

Design an Application Gateway configuration for an e-commerce site with these requirements:

  1. Main site at / serves from a pool of 3 web servers.
  2. /api/* routes to 2 API servers.
  3. /static/* routes to a storage-backed pool.
  4. /admin/* should only be accessible from the internal network (10.0.0.0/8).
  5. Draw or document the complete configuration: listeners, rules, backend pools, and health probes.
  6. Explain: How would you add SSL termination to this setup?
Show Hint
You would need 4 backend pools (web, api, storage, admin), a path-based routing rule with path maps for each URL pattern, and health probes tailored to each backend type. For the /admin/* restriction, use an NSG on the Application Gateway subnet or WAF custom rules. For SSL termination, upload a PFX certificate to the Application Gateway, configure an HTTPS listener on port 443, and optionally add an HTTP-to-HTTPS redirect rule on port 80.
