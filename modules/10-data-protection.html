Lab 10 — Implement Data Protection

1

Provision Infrastructure via Template

▼

Deploy the lab infrastructure using an ARM template to create the virtual machines needed for backup and replication exercises.

  1. Navigate to Deploy a custom template in the Azure Portal.
  2. Select Build your own template in the editor and load the template file:
    Template az104-10-vms-edge-template.json
  3. Configure the deployment parameters:
    SettingValue
    Resource Groupaz104-rg-region1 (create new)
    RegionEast US
    Usernamelocaladmin
  4. Click Review + create, then Create.
  5. Wait for the deployment to complete before proceeding.
Note ARM templates allow repeatable infrastructure deployments. This template provisions the VMs, networking, and storage needed for the backup and disaster recovery tasks.
2

Create Recovery Services Vault

▼

Create a Recovery Services vault to store backup data and manage backup policies.

  1. Search for and select Recovery Services vaults, then click + Create.
  2. Configure the vault:
    SettingValue
    Resource Groupaz104-rg-region1
    Vault nameaz104-rsv-region1
    RegionEast US
  3. Click Review + create, then Create.
  4. Once deployed, navigate to the vault and open the Properties blade.
  5. Under Backup Configuration, review the storage replication type (default: Geo-redundant).
  6. Under Security Settings, verify the soft delete retention period is set to 14 days.
Tip Geo-redundant storage replicates backup data to a paired Azure region, providing protection against regional outages. For cost savings in non-critical environments, you can switch to Locally-redundant storage — but only before the first backup is configured.
3

Configure VM-Level Backup

▼

Configure a backup policy and enable backup for a virtual machine.

  1. In the az104-rsv-region1 vault, click + Backup.
  2. Set the backup goal:
    SettingValue
    Where is your workload running?Azure
    What do you want to back up?Virtual machine
  3. Click Backup to proceed to the policy configuration.
  4. Create a new backup policy:
    SettingValue
    Policy nameaz104-backup
    FrequencyDaily
    Time12:00 AM
    Instant restore snapshot retention2 days
  5. Under Virtual Machines, click Add and select az104-10-vm0.
  6. Click Enable Backup.
  7. Return to the vault → Backup items → Azure Virtual Machine.
  8. Select az104-10-vm0 and click Backup now to trigger an on-demand backup.
Note Instant restore snapshots are stored locally for fast recovery. The snapshot is later transferred to the vault storage. Keeping 2 days of snapshots balances fast restore capability with storage cost.
4

Monitor Azure Backup

▼

Configure diagnostic settings to monitor backup operations and review job status.

  1. Create a storage account for diagnostics:
    SettingValue
    Resource Groupaz104-rg-region1
    Name(unique storage account name)
    RegionEast US
  2. Navigate to the az104-rsv-region1 vault → Diagnostic settings.
  3. Click + Add diagnostic setting and configure:
    SettingValue
    Diagnostic setting nameaz104-backup-diagnostics
    CategoriesBackup reporting data, Backup job data, Backup alert data, Site recovery events
    DestinationArchive to a storage account
    Storage account(select the account created above)
  4. Click Save.
  5. Return to the vault and navigate to Backup Jobs to review the status of the on-demand backup triggered earlier.
Tip For production environments, consider sending diagnostic data to a Log Analytics workspace instead of a storage account. This enables rich querying with Kusto (KQL) and integration with Azure Monitor workbooks for backup reporting.
5

Enable VM Replication

▼

Configure Azure Site Recovery to replicate a VM to a secondary region for disaster recovery.

  1. Create a second Recovery Services vault in a different region:
    SettingValue
    Resource Groupaz104-rg-region2 (create new)
    Vault nameaz104-rsv-region2
    RegionWest US
  2. Navigate to az104-10-vm0 in the portal.
  3. Under Operations, select Disaster recovery.
  4. Configure the replication settings:
    SettingValue
    Target regionWest US
  5. Click Review + Start replication, then Start replication.
  6. Wait 10–15 minutes for the initial replication to complete.
  7. Return to the VM's Disaster recovery blade and verify the replication health shows Healthy.
Note Azure Site Recovery continuously replicates VM data to the target region. During a regional outage, you can fail over to the secondary region with minimal data loss. The target vault (az104-rsv-region2) must be in the same region as the failover destination.

Key Takeaways

  • Azure Backup provides simple, secure backup and recovery for Azure workloads.
  • Backup policies control the frequency of backups and retention of recovery points.
  • Azure Site Recovery replicates workloads to a secondary region for disaster recovery.
  • Recovery Services vaults centralize backup storage and management.
  • Soft delete protects against accidental deletion of backup data for 14 days.

Azure Files Backup Configuration

Restore Procedures

▼

Walk through the VM restore options without executing a restore:

  1. Navigate to a backed-up VM in the Recovery Services vault.
  2. Click Restore VM and review the available recovery points.
  3. Explain the restore options:
    • Create new VM: Deploy a new VM from the recovery point.
    • Replace existing: Swap the current VM's disks with the backed-up versions.
    • Restore disks: Restore disks to a storage account for manual attachment.
  4. Demonstrate File Recovery: mount a recovery point as a drive to recover individual files.
Note "Replace existing" requires the original VM to still exist. If the VM has been deleted, use "Create new" instead.

Exercises

Easy

Exercise 1: Backup Policy Review

  1. Navigate to a Recovery Services vault.
  2. Open Backup policies and review the default policy.
  3. Document the following: schedule frequency, retention for daily, weekly, monthly, and yearly backups.
  4. What is the maximum retention period for daily backups?
Show Hint
The default policy schedules daily backups at a specific time with 30-day retention. The maximum retention period for daily backups is 9999 days (approximately 27 years). Weekly, monthly, and yearly retention points can be configured separately for long-term compliance needs.
Medium

Exercise 2: Backup and Restore Simulation

  1. Create a Recovery Services vault and configure backup for a virtual machine.
  2. Trigger an on-demand backup and wait for it to complete.
  3. Navigate to the backup item and select Restore VM.
  4. Review the two restore options: Create new VM vs Replace existing.
  5. Document the steps required for each option but DO NOT execute the restore (to save time and cost).
Show Hint
"Replace existing" requires the original VM to still exist — it swaps the VM's disks with the backed-up versions. "Create new" deploys a fresh VM from the backup recovery point. For production scenarios, "Restore disks" gives you the most control by restoring disks to a storage account for manual attachment.
Hard

Exercise 3: Disaster Recovery Architecture

Design a DR strategy for a 3-tier application (web tier, application tier, database tier).

  1. Identify the primary region (East US) and secondary region (West US).
  2. Configure Site Recovery replication for the web and application tier VMs.
  3. For the database tier, explain why Site Recovery alone is not sufficient.
  4. Document RPO (Recovery Point Objective) and RTO (Recovery Time Objective) targets for each tier.
  5. Create a failover runbook: What steps happen during failover? In what order?
  6. Explain the difference between planned failover and unplanned failover.
Show Hint
Databases need their own replication mechanism (e.g., SQL Always On Availability Groups, Cosmos DB multi-region writes) because Site Recovery handles compute replication, not data-layer consistency. Planned failover ensures zero data loss by synchronizing all pending changes before switching over. Unplanned failover may result in data loss up to the last recovery point (RPO). A typical runbook order: (1) fail over database, (2) fail over app tier, (3) fail over web tier, (4) update DNS/Traffic Manager.
