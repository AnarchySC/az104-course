Lab 11 — Implement Monitoring

Estimated time: 40 minutes

1

Provision Infrastructure

▼

Deploy the lab infrastructure using an ARM template and enable VM Insights for monitoring.

  1. Deploy the az104-11-vm-template.json ARM template:
    SettingValue

  2. Wait for the deployment to complete.
  3. Navigate to Monitor and enable VM Insights for the deployed virtual machine.
Note VM Insights provides detailed performance and health monitoring for your VMs, including dependency mapping and performance trending.
2

Create Alert

▼

Create an alert rule that fires when a virtual machine is deleted.

  1. Navigate to Monitor → Alerts → Create alert rule.
  2. Set the Scope to your subscription (all VMs).
  3. On the Condition tab, search for and select the signal:
    SettingValue
    SignalDelete Virtual Machine (Virtual Machines)
  4. Continue to the Actions tab (configured in Task 3).
Tip Activity log alerts monitor control plane operations (create, delete, update). They fire based on events, not metrics, and do not require a Log Analytics workspace.
3

Configure Action Group

▼

Create an action group to define who gets notified when the alert fires.

  1. On the Actions tab of the alert rule, click Create action group:
    SettingValue
    Action group nameAlert the operations team
    Short nameAlertOpsTeam
    Resource Groupaz104-rg11
  2. On the Notifications tab:
    SettingValue
    Notification typeEmail
    NameVM was deleted
  3. Complete the action group creation.
  4. Set the alert rule name to VM was deleted.
  5. Click Review + create, then Create.
Note Action groups can include multiple notification types: email, SMS, push notifications, voice calls, and automated actions like Azure Functions, Logic Apps, or webhooks.
4

Trigger and Validate Alert

▼

Delete a VM to trigger the alert and verify the notification is received.

  1. Navigate to az104-vm0 and click Delete.
  2. Check the Apply force delete option and confirm deletion.
  3. Check your email for a notification from azure-noreply@microsoft.com.
  4. Navigate to Monitor → Alerts and verify 3 verbose alerts appear.
Tip Alert notifications may take a few minutes to arrive. Refresh the Alerts blade to see updated counts. Verbose (Sev 4) alerts are used for informational purposes.
5

Configure Alert Processing Rule

▼

Create a rule to suppress alert notifications during a planned maintenance window.

  1. Navigate to Monitor → Alerts → Alert processing rules.
  2. Click + Create and configure:
    SettingValue
    ActionSuppress notifications
    Schedule start10 PM tonight
    Schedule end7 AM tomorrow
    Rule namePlanned Maintenance
  3. Click Review + create, then Create.
Note Alert processing rules apply to already-fired alerts. They can suppress notifications or add action groups without modifying the original alert rules. This is ideal for planned maintenance windows.
6

Azure Monitor Log Queries

▼

Use KQL (Kusto Query Language) to query Azure Monitor logs.

  1. Navigate to Monitor → Logs and set the scope to your subscription.
  2. Run the pre-built Count heartbeats query from the query library.
  3. Switch to KQL mode for direct query editing.
  4. Run the following custom query to chart CPU utilization:
    KQL InsightsMetrics | where TimeGenerated > ago(1h) | where Name == "UtilizationPercentage" | summarize avg(Val) by bin(TimeGenerated, 5m), Computer | render timechart
  5. Review the resulting time chart showing CPU utilization per VM over the last hour.
Tip KQL processes queries top-to-bottom like a pipeline. Each line filters or transforms the data from the previous line. Use render as the final operator to visualize results.

Key Takeaways

  • Alerts enable early issue detection before user impact.
  • Action groups define notification recipients and automated responses.
  • Alert processing rules suppress notifications during planned maintenance.
  • KQL (Kusto Query Language) queries Azure Monitor logs for deep analysis.
  • Activity logs track all resource operations across your subscription.

Azure Monitor Alerts: Scope, Signals, Conditions, Severity

▼

Walk through creating alerts in Azure Monitor, demonstrating:

  1. Scope: How to target alerts at subscription, resource group, or individual resource level.
  2. Signal types: Metrics (CPU, memory), activity log (create, delete), and log search (KQL-based).

Note Dynamic thresholds use machine learning to automatically determine normal behavior patterns and only alert on true anomalies, reducing alert fatigue.

KQL Syntax Fundamentals

▼

Teach the core KQL operators through live examples:

  1. where — filter rows by condition.
  2. summarize — aggregate data (count, avg, sum, max, min).
  3. project — select and rename columns.
  4. extend — add calculated columns.
  5. bin() — group time values into intervals for charting.
  6. render — visualize results as timechart, barchart, piechart.
  7. Show how the pipe (|) operator chains commands in a top-to-bottom pipeline.
Tip Start KQL queries with the table name, then filter aggressively with where before aggregating. This reduces the data processed and speeds up queries.

Exercises

Easy

Exercise 1: Alert Exploration

  1. Navigate to Monitor → Alerts.
  2. Review any existing alerts and their severity levels (0–4).
  3. Check the Alert processing rules blade.
  4. Document: What are the 5 severity levels and when would you use each?
Show Hint
Sev 0 = Critical (service down), Sev 1 = Error (functionality impacted), Sev 2 = Warning (potential issue), Sev 3 = Informational (awareness), Sev 4 = Verbose (diagnostic detail). Match severity to business impact.
Medium

Exercise 2: KQL Query Practice

  1. Open Monitor → Logs and set scope to your subscription.
  2. Run these queries and document the results:
    • Heartbeat | summarize count() by Computer — which VMs are reporting?
    • AzureActivity | where OperationNameValue contains "delete" | project TimeGenerated, Caller, OperationNameValue — who deleted what?
    • Perf | where CounterName == "% Processor Time" | summarize avg(CounterValue) by Computer — average CPU per VM
  3. Modify one query to filter to the last 24 hours only.
Show Hint
Use | where TimeGenerated > ago(24h) to filter by time. KQL processes top-to-bottom like a pipeline. Place the time filter early in the query for best performance.
Hard

Exercise 3: Full Monitoring Solution Design

  1. Design a monitoring strategy for a production web application.
  2. Create alerts for:
    • VM CPU > 80%
    • Disk space < 20%
    • HTTP 5xx errors > 10/min
  3. Configure action groups: email to ops team, SMS to on-call engineer.
  4. Create an alert processing rule: suppress non-critical alerts during weekly maintenance window (Sunday 2–4 AM).
  5. Write KQL queries to build a dashboard showing: request rate, error rate, response time, and CPU utilization.
  6. Document: How would you integrate this with PagerDuty or ServiceNow?
Show Hint
Azure Monitor supports webhooks in action groups, which can trigger PagerDuty incidents or ServiceNow tickets via their respective APIs. Use separate action groups for different escalation tiers (email for warnings, SMS+webhook for critical).
