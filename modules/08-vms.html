Lab 08 — Manage Virtual Machines

Estimated time: 50 minutes

1

Deploy Zone-Resilient VMs

▼

Create two virtual machines deployed across availability zones for high availability.

  1. Navigate to Virtual Machines and click + Create → Azure virtual machine.
  2. Configure the first VM:
    SettingValue
    Resource Groupaz104-rg8 (create new)
    Nameaz104-vm1
    Availability ZoneZone 1
    ImageWindows Server 2025
    SizeStandard_D2s_v3
  3. On the Disks tab:
    SettingValue
    OS disk typePremium SSD
    Delete with VMChecked
  4. On the Networking tab, set Public inbound ports to None.
  5. On the Management tab, disable Boot diagnostics.
  6. Click Review + create, then Create.
  7. Repeat to create az104-vm2 with the same settings but in Availability Zone 2.
Note Deploying VMs across availability zones provides 99.99% uptime SLA. Each zone is a separate physical datacenter within the same Azure region.
2

Manage Compute and Storage Scaling

▼

Resize a VM and manage data disks including attaching, detaching, and upgrading.

  1. Navigate to az104-vm1 → Size blade.
  2. Resize the VM to Standard_D2ds_v4 and click Resize.
  3. Navigate to the Disks blade and click + Create and attach a new disk:
    SettingValue
    Disk namevm1-disk1
    Size32 GB
    Storage typeStandard HDD
  4. Click Apply to attach the disk.
  5. Return to the Disks blade and detach vm1-disk1 by clicking the X icon, then Apply.
  6. Search for the vm1-disk1 disk resource, open it, and navigate to Size + performance.
  7. Upgrade the storage type to Standard SSD and click Save.
  8. Return to az104-vm1 → Disks and re-attach the upgraded disk.
Tip You can change a disk's SKU (HDD, SSD, Premium SSD) only while it is unattached. Detach first, upgrade, then re-attach.
3

Create VM Scale Set

▼

Deploy a VM Scale Set with load balancing and a network security group.

  1. Navigate to Virtual machine scale sets and click + Create.
  2. Configure the basics:
    SettingValue
    Resource Groupaz104-rg8
    Namevmss1
    Availability ZonesZones 1, 2, 3
    Orchestration modeUniform
    ImageWindows Server 2025
    SizeStandard_D2s_v3
  3. On the Networking tab, configure the virtual network:
    SettingValue
    Virtual networkvmss-vnet (create new)
    Address range10.82.0.0/20
    Subnet namesubnet0
    Subnet range10.82.0.0/24
  4. Edit the network interface to create a new NSG named vmss1-nsg.
  5. Add an inbound rule to the NSG:
    SettingValue
    NameAllowHTTP
    Destination port80
    ProtocolTCP
    ActionAllow
    Priority1010
  6. Under Load balancing, select Azure load balancer and create a new one named vmss-lb.
  7. Click Review + create, then Create.
Note Uniform orchestration treats all instances identically — ideal for stateless workloads. Flexible orchestration allows mixing VM sizes and is better for heterogeneous workloads.
4

Scale VM Scale Sets

▼

Configure custom autoscale rules based on CPU metrics.

  1. Navigate to vmss1 → Scaling blade.
  2. Select Custom autoscale.
  3. Set the Scale mode to Scale based on a metric.
  4. Add a scale-out rule:
    SettingValue
    MetricPercentage CPU
    OperatorGreater than
    Threshold70%
    Duration10 minutes
    OperationIncrease percent by
    Value50%
  5. Add a scale-in rule:
    SettingValue
    MetricPercentage CPU
    OperatorLess than
    Threshold30%
    Duration10 minutes
    OperationDecrease percent by
    Value50%
  6. Set the instance limits:
    SettingValue
    Minimum2
    Maximum10
    Default2
  7. Click Save.
Tip Always configure both scale-out and scale-in rules. Without a scale-in rule, you may end up paying for idle instances that never get removed.
5

Create VM with PowerShell (Optional)

▼

Use Azure PowerShell to create a virtual machine from the command line.

  1. Open Azure Cloud Shell and select PowerShell.
  2. Run the following command to create a VM:
    PowerShell New-AzVm -ResourceGroupName 'az104-rg8' -Name 'myPSVM' -Location 'East US' -Image 'Win2019Datacenter' -Zone '1' -Size 'Standard_D2s_v3'
  3. When prompted, enter a username and password for the VM admin account.
  4. Once deployment completes, verify the VM appears in the az104-rg8 resource group.
Note PowerShell is ideal for scripted or repeatable deployments. The New-AzVm cmdlet creates a VM along with all required resources (NIC, disk, NSG) automatically.
6

Create VM with CLI (Optional)

▼

Use Azure CLI to create a Linux virtual machine.

  1. Open Azure Cloud Shell and select Bash.
  2. Run the following command to create a VM:
    CLI az vm create --name myCLIVM --resource-group az104-rg8 --image Ubuntu2204 --admin-username localadmin --generate-ssh-keys
  3. Wait for the deployment to complete. The output will include the public IP address.
  4. Verify the VM appears in the az104-rg8 resource group.
Tip The --generate-ssh-keys flag automatically creates an SSH key pair if one doesn't exist. The private key is stored in ~/.ssh/ in your Cloud Shell session.

Key Takeaways

  • Virtual machines provide on-demand, scalable computing resources in Azure.
  • Both vertical scaling (resizing VM size) and horizontal scaling (adding instances) are available.
  • VM Scale Sets enable load-balanced group management with autoscale rules.
  • Availability zones support 99.99% uptime SLA by distributing VMs across physical datacenters.

1

VM Creation in Portal (Windows/Linux)

▼

Walk through creating virtual machines in the Azure Portal, demonstrating:

  1. Choosing between Windows Server and Linux (Ubuntu) images.
  2. Selecting a VM size based on workload requirements (B-series vs D-series).
  3. Configuring authentication — password for Windows, SSH key for Linux.
  4. Reviewing disk options: OS disk type, data disks, delete-with-VM setting.

Tip Use the See all sizes link to filter by family, vCPUs, RAM, and cost. Bookmark sizes you use frequently.
2

Connection Methods: RDP, SSH, Bastion

▼

Demonstrate the three main ways to connect to a VM:

  1. RDP: Download the RDP file from the VM's Connect blade. Requires port 3389 open in the NSG.
  2. SSH: Connect using ssh adminuser@<public-ip>. Requires port 22 open in the NSG.
  3. Bastion: Connect through the browser without a public IP. Requires AzureBastionSubnet in the VNet.
  4. Compare security implications: Bastion eliminates the need for public IP exposure and open RDP/SSH ports.
Note Azure Bastion provides secure RDP/SSH over TLS directly from the Azure Portal, without exposing VM ports to the public internet.
3

VM Scale Sets: Uniform vs Flexible Orchestration

▼

Compare the two orchestration modes available for VM Scale Sets:

  1. Uniform: All instances use the same VM model and image. Best for stateless, identical workloads (web servers, workers).
  2. Flexible: Mix different VM sizes and images. Best for heterogeneous workloads or when migrating from availability sets.
  3. Show how to select orchestration mode during VMSS creation.
  4. Demonstrate the Instances blade to view running instances and their status.
  5. Show the Upgrade policy options: Automatic, Rolling, Manual.
4

Scaling Configuration: Manual and CPU-Based Autoscale

▼

Demonstrate both manual and automatic scaling options:

  1. Show manual scaling: change the instance count directly on the Scaling blade.
  2. Switch to custom autoscale and walk through the rule builder.
  3. Create a CPU-based scale-out rule: average CPU > 70% for 10 min → add 1 instance.
  4. Create a matching scale-in rule: average CPU < 30% for 10 min → remove 1 instance.
  5. Explain cooldown period and why it matters (prevents flapping).
  6. Show the Run history tab to review past scaling events.
Tip Always set a cooldown period (default 5 minutes) to allow metrics to stabilize after a scaling event before triggering another.

Exercises

Easy

Exercise 1: VM Size Comparison

  1. When creating a VM, explore the Size picker using See all sizes.
  2. Compare at least 3 VM families: B-series (burstable), D-series (general purpose), E-series (memory-optimized).
  3. Document vCPUs, RAM, temp storage, and estimated cost for one size in each family.
  4. Which family would you choose for a web server? A database? A dev/test environment?
Show Hint
B-series is cheapest for dev/test (burstable CPU credits). D-series is the general-purpose workhorse for production web servers. E-series has a high memory-to-CPU ratio, making it ideal for databases and in-memory caching.
Medium

Exercise 2: Availability Architecture

  1. Create a VM and note which availability zone it is deployed in.
  2. Review the difference between: Availability Zones, Availability Sets, and VM Scale Sets.
  3. Create a table comparing: fault domains, update domains, SLA, and use cases for each option.
  4. If you need 99.99% SLA, which option do you choose and why?
  5. What is the SLA for a single VM with Premium SSD?
Show Hint
Single VM with Premium SSD = 99.9% SLA. Availability Set = 99.95% SLA (2+ VMs across fault/update domains). Availability Zones = 99.99% SLA (2+ VMs across physically separate datacenters). Choose Availability Zones for the highest SLA.
Hard

Exercise 3: Autoscale Simulation

  1. Deploy a VM Scale Set with custom autoscale (CPU > 70% = scale out).
  2. SSH or RDP into an instance.
  3. Generate CPU load (e.g., run a stress test or busy loop).
  4. Monitor the scale set's Instances blade — watch for new instances appearing.
  5. Stop the load and observe scale-in behavior.
  6. Review the Activity Log for scaling events.
Show Hint
On Linux, use stress --cpu 4 --timeout 600 to generate CPU load. On Windows, use a PowerShell busy loop: while($true){}. Scaling may take 5–10 minutes to trigger due to metric aggregation and cooldown periods.
